Why This Exists

소개 페이지가 아니라 데이터 운영판

참고 사이트의 어두운 톤, 큰 첫 화면, 아키텍처 설명, 탐색기를 반영하되 OpenFin의 실제 manifest와 MCP 사용 흐름을 중심으로 내용을 다시 구성했습니다.

기존 금융 정보의 문제

  • 세금, 복지, 카드 혜택, 은행 금리, 보험 조건이 서로 다른 기관과 파일에 흩어져 있습니다.
  • 상품명, 우대조건, 한도, 기준일은 자주 바뀌므로 앱과 AI가 같은 기준을 읽어야 합니다.
  • 원문 출처와 수집일이 빠지면 금융·세무 답변의 검증 가능성이 떨어집니다.

OpenFin에 반영한 방식

  • 5개 export를 manifest로 묶고 도메인별 item_count와 product_count를 첫 화면에 노출합니다.
  • 각 노드는 ID, 타입, 설명, provider, criteria, source_urls, 이웃 노드를 유지합니다.
  • GitHub Pages 탐색기와 Cloudflare MCP가 같은 export를 읽도록 연결했습니다.

Content Additions

추가하면 좋은 내용과 반영 방식

현재 온톨로지를 기준으로 웹과 MCP에서 바로 가치가 커지는 확장 항목입니다.

데이터 신뢰도 배지

각 노드에 공식 API, 기관 페이지, 수기 overlay, 기준일 만료 여부를 배지로 표시합니다.

반영: `source_urls`, `source_basis_dates`, `revision_status`를 상세 패널과 MCP fetch 응답에서 동일하게 노출

상품 비교 축

카드 혜택률, 예적금 금리, 대출 평균금리, 보험 수익률처럼 도메인별 비교 가능한 값을 별도 표로 빼냅니다.

반영: `criteria`와 `options`의 수치 필드를 normalize하여 product metric index 생성

사용자 시나리오 경로

청년, 소상공인, 주택 임차인, 연말정산 근로자처럼 실제 탐색 목적별 출발점을 만듭니다.

반영: `scenario.*` 노드와 `path_steps`를 추가하고 연관 노드 ID를 검증

변경 이력

금리·혜택·세제 기준이 언제 바뀌었는지 Git diff와 수집일 기준으로 비교할 수 있게 합니다.

반영: manifest version별 export hash, changed item count, ended product count를 별도 manifest에 기록
